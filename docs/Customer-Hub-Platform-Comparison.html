Customer Hub: platform comparison

A realistic assessment of building the Customer Hub experience using SharePoint, Smartsheet, or a purpose-built application.

Executive summary

The Customer Hub consolidates data from Smartsheet, HubSpot, and AI-powered document generation into a single branded portal for customers and a management dashboard for ESM staff. This document compares what the custom application delivers against what SharePoint and Smartsheet can realistically provide using their native capabilities.

SharePoint and Smartsheet are strong tools for their intended purposes, but neither can replicate the hub's core differentiators: AI document generation, cross-platform data unification, magic link authentication, automatic provisioning from HubSpot, or Outlook email logging against intake records. A SharePoint or Smartsheet version would require significant manual effort per customer and would not deliver the same experience.

The application is already built and functional. The decision is not whether to invest in building it, but whether to deploy what already exists. At current team size, the hub saves an estimated 5–8 hours of SC/PM time per project through automated document generation and provisioning — time that compounds as the customer portfolio grows.

Customer impression

Implementation is often the customer's first hands-on experience with ESM after signing. The portal they interact with during this period shapes their perception of ESM as a technology partner.

Customer Hub
A branded, purpose-built portal with the customer's name, their specific timeline, their action items, and their documents. Login is a single magic link — no accounts to create, no licenses to manage. The experience signals that ESM has invested in their success.
SharePoint / Smartsheet
A generic SharePoint site or Smartsheet dashboard that looks like an internal tool shared externally. Customers see Microsoft or Smartsheet branding, standard grid layouts, and may need additional licenses just to view their own project data. The experience feels like an afterthought.

Risks and considerations

The custom application approach has real tradeoffs that should be weighed against the benefits.

Maintenance responsibility
A custom app requires ongoing maintenance — dependency updates, API changes from Smartsheet/HubSpot, and bug fixes. SharePoint and Smartsheet handle this as part of their SaaS service.
Mitigation: The app is stateless with no database. Maintenance is limited to Node.js updates and occasional API changes.
Knowledge concentration
The codebase was built by one person. If that person is unavailable, updates and troubleshooting require someone with Next.js/TypeScript knowledge.
Mitigation: Codebase is documented (PM guide + technical hosting guide). Standard Next.js patterns — any TypeScript developer can maintain it.
AI output quality
AI-generated documents require SC review before delivery. Output quality depends on the completeness of source data in Smartsheet and HubSpot. Poor input data produces poor documents.
Mitigation: AI provides a strong first draft, not a final product. SCs review and edit before sharing with customers.
API cost variability
Anthropic API costs are usage-based. High document generation volume or complex extractions could push costs above the $50/month estimate.
Mitigation: Usage can be monitored in the Anthropic Console. Cost caps can be set. Switching to a lighter model reduces costs further.

Security and data residency

The Customer Hub does not introduce a new data store. All customer data remains in the systems ESM already manages.

ConcernHow the Customer Hub handles it
Customer data storageNo database. All project data lives in Smartsheet and HubSpot. The hub reads it on demand and does not persist copies.
Uploaded filesCustomer uploads are processed in memory and stored as Smartsheet attachments. No files are retained on the application server.
Generated documentsAI-generated documents are uploaded to the customer's Smartsheet workspace and served via temporary download URLs. Nothing is stored locally.
Authentication (ESM staff)NextAuth with JWT sessions. Passwords stored as bcrypt hashes in a config file on the server. Can be migrated to Azure AD/SSO in a future phase.
Authentication (customers)Magic link tokens sent via email. One-time use, 24-hour expiry. Sessions are signed JWTs with 30-day expiry. No customer passwords or credentials stored.
Data in transitAll connections are HTTPS — browser to app, app to external APIs. TLS terminated at the Azure App Service or reverse proxy layer.
AI processingDocument content is sent to Anthropic’s Claude API for generation. Anthropic does not train on API inputs and deletes them within 30 days per their data policy.
API credentialsStored as environment variables on the Azure App Service, not in code or version control. Rotatable without code changes.

The application is stateless by design. If the server is lost or rebuilt, no customer data is affected — only the server configuration (environment variables and a small JSON config file) needs to be restored.

Recommendation

SharePoint and Smartsheet are effective tools for document storage and project tracking respectively, but neither can deliver a unified customer experience that combines live data from multiple sources with AI-powered document generation.

The Customer Hub saves an estimated 5–8 SC/PM hours per project, presents a professional branded experience to customers during their critical first engagement with ESM, and scales without proportional increases in manual effort. The application is already built and documented — deploying it requires only an Azure App Service instance and production API keys.

The risks are real but manageable: maintenance is lightweight (no database, standard framework), AI output requires human review, and the codebase is documented for handoff. At $28–76/month in new costs, the investment is modest relative to the time it returns.
